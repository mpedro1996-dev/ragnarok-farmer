# Especificação: CRUD de Personagem

## Resumo

`Character` será o terceiro CRUD administrativo do projeto. A funcionalidade viverá dentro da própria aplicação Next.js, sem separar SPA e API, e a primeira entrega será uma área interna acessível em `/admin/characters`.

O objetivo do v1 é permitir criar, listar, editar e excluir personagens de forma definitiva, usando SQLite como banco local e uma camada ORM com migrações. A interface deve seguir os princípios de `DESIGN.MD`, priorizando legibilidade, consistência visual, estados acessíveis e abordagem mobile-first.

## Escopo do Personagem

Cada personagem deve possuir os seguintes campos:

- `id`: identificador interno
- `name`: obrigatório, texto único
- `level`: obrigatório, inteiro positivo
- `classId`: obrigatório, inteiro fixo de enum
- timestamps podem existir conforme a camada de persistência

### Regras de validação

- `name` deve ser salvo sem espaços extras nas pontas
- `name` não pode ser vazio
- `name` deve ser único
- `level` deve ser inteiro maior que zero
- `classId` deve existir na enum fixa da v1
- a v1 não inclui vínculos com instâncias, itens, contas ou outras entidades

## Enum de classes

As classes devem ser tratadas como lista fixa local, armazenando `classId` no contrato e exibindo `label` na interface.

## Experiência da Interface

A rota `/admin/characters` deve concentrar a experiência administrativa do v1 em uma única página.

### Listagem

A tela deve exibir uma tabela ou lista administrativa com, no mínimo:

- Nome
- Nível
- Classe
- Ações de editar e excluir

A listagem deve suportar:

- busca por nome
- ordenação por nome
- ordenação por nível
- ordenação por classe

### Criação e edição

Criação e edição devem acontecer na mesma página, usando modal de formulário.

O formulário deve conter:

- Nome
- Nível
- combobox de Classe alimentada por enum fixa local

O usuário deve receber feedback claro para:

- campos inválidos
- sucesso ao salvar
- erro de duplicidade
- erro de persistência
- tentativa de usar `classId` fora da enum

### Exclusão

A exclusão no v1 será definitiva. Antes de remover, a interface deve exigir confirmação explícita do usuário.

## Backend e Persistência

A API deve ficar dentro do mesmo projeto Next.js, usando route handlers internos em `/api`.

### Endpoints

- `GET /api/characters`
  - aceita `search`, `sortBy` e `sortOrder`
  - retorna a lista usada pela área administrativa
  - inclui `classLabel` no payload

- `POST /api/characters`
  - cria personagem com `name`, `level` e `classId`
  - rejeita nome duplicado
  - rejeita `level` inválido
  - rejeita `classId` fora da enum fixa

- `PUT /api/characters/:id`
  - atualiza os mesmos campos com as mesmas validações

- `DELETE /api/characters/:id`
  - remove o personagem de forma permanente

### Banco

- usar SQLite como persistência local padrão
- usar ORM com suporte a migrações
- persistir `Character` em tabela própria
- indexar `name`, `level` e `classId`

## Cenários de aceite

- criar personagem com nome válido, nível válido e classe válida
- impedir criação com nome duplicado
- impedir `level` inválido
- impedir `classId` inválido
- editar nome, nível e classe com sucesso
- buscar personagem por nome
- ordenar por nome, nível e classe
- excluir personagem de forma definitiva após confirmação
- manter página, API e banco dentro do mesmo projeto Next.js
