# Especificação: CRUD de Instância PvE

## Resumo

`Instance` será o segundo CRUD administrativo do projeto. A funcionalidade viverá dentro da própria aplicação Next.js, sem separar SPA e API, e a primeira entrega será uma área interna acessível em `/admin/instances`.

O objetivo do v1 é permitir criar, listar, editar e excluir instâncias PvE farmáveis de forma definitiva, usando SQLite como banco local e uma camada ORM com migrações. A interface deve seguir os princípios de `DESIGN.MD`, priorizando legibilidade, consistência visual, estados acessíveis e abordagem mobile-first.

## Escopo da Instância

Cada instância deve possuir os seguintes campos:

- `id`: identificador interno
- `name`: obrigatório, texto único
- `minimumLevel`: obrigatório, inteiro positivo
- `cooldownDays`: obrigatório, inteiro maior ou igual a zero
- `items`: relação simples com itens já cadastrados
- timestamps podem existir conforme a camada de persistência

### Regras de validação

- `name` deve ser salvo sem espaços extras nas pontas
- `name` não pode ser vazio
- `name` deve ser único
- `minimumLevel` deve ser inteiro maior que zero
- `cooldownDays` deve ser inteiro maior ou igual a zero
- a instância pode ser salva sem itens relacionados
- o mesmo item pode aparecer em várias instâncias
- o mesmo item não pode ser vinculado duas vezes à mesma instância

## Experiência da Interface

A rota `/admin/instances` deve concentrar a experiência administrativa do v1 em uma única página.

### Listagem

A tela deve exibir uma tabela ou lista administrativa com, no mínimo:

- Nome
- Nível mínimo
- Tempo de espera em dias
- Quantidade de itens vinculados
- Ações de editar e excluir

A listagem deve suportar:

- busca por nome
- ordenação por nome
- ordenação por nível mínimo
- ordenação por tempo de espera em dias

### Criação e edição

Criação e edição devem acontecer na mesma página, usando modal de formulário.

O formulário deve conter:

- Nome
- Nível mínimo
- Tempo de espera em dias
- bloco de gerenciamento de itens

O usuário deve receber feedback claro para:

- campos inválidos
- sucesso ao salvar
- erro de duplicidade
- erro de persistência
- tentativa de vincular item inexistente

### Gerenciamento de itens

O vínculo com itens deve usar duas tabelas:

- tabela de itens já adicionados, com ícone, nome e botão de remover
- tabela de itens disponíveis, com ícone, nome e botão de adicionar

A tabela de itens disponíveis deve permitir busca por nome.

Se não houver itens cadastrados, a interface deve mostrar um empty state orientando o usuário a criar itens primeiro em `/admin/items`.

Se um item tiver `divinePrideId`, o ícone deve ser exibido pela URL:

`https://static.divine-pride.net/images/items/item/{divinePrideId}.png`

Se `divinePrideId` estiver vazio, a interface deve mostrar fallback seguro, sem quebrar layout nem carregar imagem inválida.

### Exclusão

A exclusão no v1 será definitiva. Antes de remover, a interface deve exigir confirmação explícita do usuário.

## Backend e Persistência

A API deve ficar dentro do mesmo projeto Next.js, usando route handlers internos em `/api`.

### Endpoints

- `GET /api/instances`
  - aceita `search`, `sortBy` e `sortOrder`
  - retorna a lista usada pela área administrativa
  - inclui `itemCount` e uma lista leve de itens vinculados

- `POST /api/instances`
  - cria instância com `name`, `minimumLevel`, `cooldownDays` e `itemIds`
  - rejeita nome duplicado
  - rejeita IDs de item inexistentes
  - rejeita item repetido no mesmo payload

- `PUT /api/instances/:id`
  - atualiza os mesmos campos com as mesmas validações

- `DELETE /api/instances/:id`
  - remove a instância de forma permanente
  - remove apenas os vínculos com itens, nunca os itens em si

### Banco

- usar SQLite como persistência local padrão
- usar ORM com suporte a migrações
- persistir `Instance` em tabela própria
- persistir a relação com itens em tabela de junção dedicada
- garantir unicidade por par `instanceId` + `itemId`

## Compatibilidade com Item

Se um item estiver vinculado a uma ou mais instâncias, o CRUD de itens deve bloquear sua exclusão e informar que o vínculo precisa ser removido antes.

## Cenários de aceite

- criar instância com nome válido, nível mínimo válido, cooldown `0` e sem itens
- criar instância com um ou mais itens vinculados
- impedir criação com nome duplicado
- impedir `minimumLevel` inválido
- impedir `cooldownDays` negativo ou não inteiro
- impedir vínculo com item inexistente
- editar instância e adicionar itens
- editar instância e remover itens
- buscar instância por nome
- ordenar por nome, nível mínimo e tempo de espera em dias
- mostrar contagem correta de itens na listagem
- excluir instância e preservar os itens
- bloquear exclusão de item que ainda esteja vinculado a alguma instância
- exibir fallback visual quando item vinculado não tiver `divinePrideId`
- manter página, API e banco dentro do mesmo projeto Next.js
