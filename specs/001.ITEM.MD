# Especificação: CRUD de Item

## Resumo

`Item` será o primeiro CRUD administrativo do projeto. A funcionalidade viverá dentro da própria aplicação Next.js, sem separar SPA e API, e a primeira entrega será uma área interna acessível em `/admin/items`.

O objetivo do v1 é permitir criar, listar, editar e excluir itens de forma definitiva, usando SQLite como banco local e uma camada ORM com migrações. A interface deve seguir os princípios de `DESIGN.MD`, priorizando legibilidade, consistência visual, estados acessíveis e abordagem mobile-first.

## Escopo do Item

Cada item deve possuir os seguintes campos:

- `id`: identificador interno
- `name`: obrigatório, texto único
- `averageZenny`: obrigatório, inteiro positivo
- `divinePrideId`: opcional, inteiro positivo quando informado
- `isSoldToNpc`: obrigatório na persistência, booleano com default `false`
- timestamps podem existir conforme a camada de persistência

### Regras de validação

- `name` deve ser salvo sem espaços extras nas pontas
- `name` não pode ser vazio
- `name` deve ser único
- `averageZenny` deve ser inteiro maior que zero
- `divinePrideId` pode ser nulo
- quando `divinePrideId` existir, ele deve ser inteiro maior que zero
- `isSoldToNpc` deve ser salvo como booleano
- checkbox desmarcado significa `false`
- checkbox marcado significa `true`

## Experiência da Interface

A rota `/admin/items` deve concentrar a experiência administrativa do v1 em uma única página.

### Listagem

A tela deve exibir uma tabela ou lista administrativa com, no mínimo:

- Ícone
- Nome
- Valor médio em zenny
- ID Divine Pride
- Venda NPC
- Ações de editar e excluir

A listagem deve suportar:

- busca por nome
- ordenação por nome
- ordenação por valor médio em zenny

Se `divinePrideId` estiver preenchido, o ícone deve ser exibido pela URL:

`https://static.divine-pride.net/images/items/item/{divinePrideId}.png`

Se `divinePrideId` estiver vazio, a interface deve mostrar um fallback seguro, sem quebrar layout nem carregar imagem inválida.

### Criação e edição

Criação e edição devem acontecer na mesma página, usando modal de formulário.

O formulário deve conter:

- Nome
- Valor médio em zenny
- ID Divine Pride
- Venda para NPC

O usuário deve receber feedback claro para:

- campos inválidos
- sucesso ao salvar
- erro de duplicidade
- erro de persistência

### Exclusão

A exclusão no v1 será definitiva. Antes de remover, a interface deve exigir confirmação explícita do usuário.

## Backend e Persistência

A API deve ficar dentro do mesmo projeto Next.js, usando route handlers internos em `/api`.

### Endpoints

- `GET /api/items`
  - aceita `search`, `sortBy` e `sortOrder`
  - retorna a lista usada pela área administrativa

- `GET /api/items/:id`
  - retorna o detalhe do item em edição
  - inclui `item` e `priceHistory[]`
  - `priceHistory` deve vir ordenado do mais recente para o mais antigo

- `POST /api/items`
  - cria item com `name`, `averageZenny`, `divinePrideId` opcional e `isSoldToNpc`
  - rejeita nome duplicado

- `PUT /api/items/:id`
  - atualiza os mesmos campos com as mesmas validações
  - cria histórico de preço apenas quando `averageZenny` mudar de fato

- `DELETE /api/items/:id`
  - remove o item de forma permanente

### Banco

- usar SQLite como persistência local padrão
- usar ORM com suporte a migrações
- a escolha exata do ORM pode ser definida na implementação, desde que suporte bem SQLite e evolução de schema

### Histórico de alteração de preço

O item deve possuir uma trilha administrativa de alterações de preço, separada da listagem principal.

- nome técnico recomendado: `ItemPriceHistory`
- relação: `Item 1:N ItemPriceHistory`
- campos mínimos:
  - `id`
  - `itemId`
  - `previousAverageZenny`
  - `nextAverageZenny`
  - `createdAt`

Regras:

- criar histórico apenas quando `averageZenny` mudar
- alterar `name`, `divinePrideId` ou `isSoldToNpc` sem mudar preço não gera histórico
- o histórico deve ser salvo na mesma transação da atualização do item
- criação de item não gera histórico inicial
- relatórios continuam usando apenas o `averageZenny` atual do item

### Exibição do histórico no modal

- o bloco de histórico aparece somente na edição
- no desktop, o modal de edição deve usar duas colunas:
  - coluna principal com o formulário
  - coluna secundária com o histórico de preço
- no mobile, o histórico pode voltar a empilhar abaixo do formulário
- cada linha deve mostrar `preço antigo -> preço novo`
- cada linha deve mostrar também a data e hora da alteração em `pt-BR`
- quando não houver registros, exibir `Nenhuma alteração de preço registrada.`
- se a lista crescer, usar altura máxima com rolagem interna sem expandir demais o modal
- a coluna do histórico deve funcionar como painel lateral administrativo, preservando leitura rápida enquanto o usuário edita os campos

## Integração com Divine Pride

No v1, a integração com Divine Pride será apenas para exibição do ícone por URL. O tooltip completo não faz parte desta primeira entrega.

O script abaixo não deve ser carregado globalmente nesta fase:

`<script type="text/javascript" src="https://www.divine-pride.net/scripts/tooltip.js"></script>`

Se o tooltip for adotado no futuro, ele deve ser carregado somente na página ou no contexto que realmente utilizar esse comportamento.

## Cenários de aceite

- criar item com campos obrigatórios válidos
- criar item sem `divinePrideId`
- criar item com `isSoldToNpc = false`
- criar item com `isSoldToNpc = true`
- impedir criação com nome duplicado
- impedir `averageZenny` inválido
- impedir `divinePrideId` inválido quando informado
- editar item existente com sucesso
- editar item alternando o status de venda para NPC
- editar item sem mudar `averageZenny` e não gerar histórico
- editar item mudando `averageZenny` e gerar histórico com preço antigo e novo
- carregar detalhe do item com `priceHistory` ordenado do mais recente para o mais antigo
- excluir item de forma definitiva após confirmação
- buscar item por nome
- ordenar por nome e por valor médio em zenny
- mostrar ícone quando houver `divinePrideId`
- mostrar fallback quando `divinePrideId` estiver ausente
- exibir o status de venda NPC na listagem administrativa
- manter página, API e banco dentro do mesmo projeto Next.js
